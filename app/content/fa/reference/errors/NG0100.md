# تغییر عبارت پس از بررسی

<docs-video src="https://www.youtube.com/embed/O47uUnJjbJc"/>

Angular زمانی `ExpressionChangedAfterItHasBeenCheckedError` ایجاد می‌کند که مقدار یک عبارت پس از پایان change detection تغییر کرده باشد. Angular این خطا را فقط در development mode ایجاد می‌کند.

در development mode،‏ Angular پس از هر اجرای change detection بررسی دیگری انجام می‌دهد تا مطمئن شود bindingها تغییر نکرده‌اند. این کار خطاهایی را پیدا می‌کند که view را در وضعیت ناسازگار باقی می‌گذارند. برای مثال، ممکن است method یا getter در هر فراخوانی مقدار متفاوتی برگرداند یا component فرزند مقادیر والد را تغییر دهد. هر یک از این موارد نشان می‌دهد change detection پایدار نشده است. Angular برای اطمینان از بازتاب درست داده در view خطا ایجاد می‌کند و از رفتار نامنظم UI یا حلقه بی‌نهایت احتمالی جلوگیری می‌کند.

این خطا معمولاً پس از افزودن عبارت‌های template یا آغاز پیاده‌سازی lifecycle hookهایی مانند `ngAfterViewInit` یا `ngOnChanges` رخ می‌دهد. همچنین هنگام کار با وضعیت loading و عملیات asynchronous یا زمانی که component فرزند bindingهای والد را تغییر می‌دهد رایج است.

## اشکال‌زدایی خطا

[source mapهای](https://developer.mozilla.org/docs/Tools/Debugger/How_to/Use_a_source_map) تولیدشده توسط CLI برای debugging بسیار مفیدند. در call stack به عقب بروید تا عبارت templateای را پیدا کنید که مقدار نمایش‌داده‌شده در خطا در آن تغییر کرده است.

مطمئن شوید پس از اجرای change detection هیچ bindingای در template تغییر نمی‌کند. این کار اغلب به‌معنای بازآرایی برای استفاده از lifecycle hook مناسب use case شماست. اگر مشکل در `ngAfterViewInit` است، راهکار پیشنهادی استفاده از constructor یا `ngOnInit` برای تنظیم مقادیر اولیه، یا استفاده از `ngAfterContentInit` برای bindingهای دیگر است.

اگر در view به methodها binding دارید، مطمئن شوید فراخوانی آن‌ها هیچ binding دیگری را در template به‌روزرسانی نمی‌کند.
